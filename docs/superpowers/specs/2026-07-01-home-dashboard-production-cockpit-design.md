# Дизайн-спека: главная страница Production Cockpit

## Контекст

Главная страница должна быть управленческим экраном производственного контура ГОКа. Пользователь открывает ее, чтобы быстро понять выполнение плана, увидеть отклонения, связать их с активами и возможными причинами, а затем перейти в детализацию.

Старый подход с крупными одинаковыми секциями "Добыча", "Минеральные ресурсы", "Обогащение" не подходит как основа главной: он показывает много KPI, но не собирает вывод. Новая главная должна быть не каталогом метрик, а cockpit-экраном: итог, динамика, причины, активы.

## Источники и роль источников

Открытые источники Polyus используются только для понимания домена, состава показателей и производственной логики. Визуальный стиль сайта Polyus не используется.

Основные источники:

- Polyus factsheet: https://polyus.com/en/investors/factsheet/
- Polyus operating mines: https://polyus.com/en/operations/operating_mines/
- Страницы активов Polyus: Olimpiada, Blagodatnoye, Natalka и другие страницы в разделе operating mines.
- Визуальные референсы: Muzli/Zajno dashboard для Shopify intelligence platform и приложенные скриншоты пользователя.

Из открытых материалов Polyus берем концептуальный набор операционных показателей:

- добытая руда;
- переработанная руда;
- производство золота;
- содержание Au;
- извлечение;
- горная масса;
- коэффициент вскрыши;
- разрез по активам/месторождениям.

Из визуальных референсов берем:

- мягкую dashboard-композицию;
- крупные округлые карточки;
- чистую типографику;
- компактные delta-pills;
- один главный график вместо множества равных блоков;
- таблицы/списки с легкими разделителями;
- светлый modern SaaS стиль как основное направление.

## Цель экрана

Главная должна отвечать на пять вопросов за первые 10-20 секунд:

1. Выполняется ли план по итоговому производственному KPI?
2. Где возникло отклонение: добыча, транспорт, переработка, качество сырья, извлечение?
3. Какой актив дает основной вклад в отклонение?
4. Есть ли объяснимая причина: простой, КИО/КТГ, ремонт, ограничение фабрики, логистика, нет данных?
5. Куда перейти, чтобы разобраться детальнее?

## Рекомендуемый подход

Рекомендуемый подход: Production Cockpit.

Главная строится вокруг производственного потока и план-факта, а не вокруг организационных разделов. Страница должна показывать outcome, динамику, отклонения, причины и активы. Детальные разделы "Добыча", "Минеральные ресурсы", "Обогащение" остаются в навигации или детализации, но не являются главной компоновочной моделью.

## Информационная иерархия

1. Итоговый статус и ключевые KPI периода.
2. План-факт по главной метрике.
3. Отклонения с причинами.
4. Производственный поток.
5. Активы/месторождения.
6. События периода и качество данных.

## Структура главной

### 1. Верхняя KPI-лента

Назначение: за один взгляд показать состояние производственного контура.

Рекомендуемые KPI:

- "Золото произведено" - главный итоговый KPI, факт / план / отклонение;
- "Руда добыта" - upstream production;
- "Руда переработана" - throughput фабрики;
- "Извлечение" - эффективность переработки;
- "Содержание Au" - качество сырья;
- "KPI в норме" - агрегированный health-индикатор.

Формат карточки:

- название метрики;
- крупное значение;
- единица измерения;
- delta-pill;
- подпись: "к плану", "к прошлой смене", "за период";
- статус по бизнес-правилу, а не только по знаку дельты.

Примеры моковых значений:

- "Золото произведено" - `8.24 тыс. унц`, `-4.1% к плану`;
- "Руда добыта" - `98.0 тыс. т`, `-6.2%`;
- "Руда переработана" - `50.5 тыс. т`, `+1.7%`;
- "Извлечение" - `84.2%`, `-1.1 п.п.`;
- "Содержание Au" - `1.42 г/т`, `-3.4%`;
- "KPI в норме" - `11/16`, `3 критично`.

### 2. Главный график "План-факт"

Назначение: показать, отклонение разовое или накапливается.

Основная метрика по умолчанию: "Золото произведено".

Переключатели:

- "Золото";
- "Руда добыта";
- "Руда переработана";
- "Извлечение";
- "Содержание Au".

Формат:

- cumulative line chart за выбранный период;
- линия плана;
- линия факта;
- зона отставания;
- маркеры дней/смен с критичными отклонениями;
- summary справа или сверху: факт, план, отклонение.

Почему это главный график:

- производство золота является итоговым бизнес-результатом;
- график помогает понять динамику разрыва;
- переключатели позволяют увидеть, где именно просадка проявилась раньше.

### 3. Блок "Отклонения и причины"

Назначение: заменить абстрактные "узкие места" на объяснимый список проблем.

Показывать top 3-5 отклонений, ранжированных по влиянию на итоговый KPI.

Структура элемента:

- актив/месторождение;
- стадия;
- метрика;
- факт / план;
- отклонение;
- причина;
- влияние;
- ссылка в детализацию.

Причины:

- простой оборудования;
- снижение КИО;
- снижение КТГ;
- ремонт;
- ограничение фабрики;
- логистическое ограничение;
- отклонение содержания Au;
- нет данных;
- причина не классифицирована.

Пример:

`Наталка · Переработка · Руда переработана 42.1 / 46.0 тыс. т · -8.5% · ограничение фабрики · влияние -0.42 тыс. унц`

Визуально:

- единый список или компактные карточки-строки;
- status-pill справа;
- причина отдельным бейджем;
- красный используется только для критичных отклонений;
- не делать каждое отклонение тяжелой отдельной карточкой, если список длинный.

### 4. Производственный поток

Назначение: показать, где в цепочке возникло отклонение.

Рекомендуемые этапы:

`Горная масса -> Руда добыта -> Транспорт -> Руда переработана -> Извлечение -> Золото`

Для каждого этапа:

- факт / план;
- отклонение;
- статус;
- худшая причина или событие;
- переход в детализацию этапа.

Важно: блок не должен быть декоративной схемой. Он полезен только если на каждом этапе есть реальные план-факт данные и статус.

Если для транспорта или ремонтов данных пока нет, этапы можно скрыть или показывать только в блоке причин.

### 5. Активы / месторождения

Назначение: показать вклад каждого актива в выполнение плана.

Формат: компактная таблица или карточки-строки, не большая сетка KPI.

Рекомендуемые колонки:

- актив;
- статус;
- золото произведено;
- руда добыта;
- руда переработана;
- извлечение;
- худшее отклонение;
- причина;
- вклад в отклонение.

Активы:

- Олимпиада;
- Благодатное;
- Наталка;
- Куранах;
- Сухой Лог;
- при необходимости Вернинское или другие активы, если они есть в контуре продукта.

Отдельный полезный виджет внутри этого блока:

- horizontal bar chart "Вклад активов в производство";
- stacked bar "Факт/план по активам";
- таблица остается основным форматом, график - поддерживающим.

### 6. События периода

Назначение: дать контекст к отклонениям, если доступны события из операционных систем.

События:

- простои;
- ремонты;
- ограничения фабрики;
- логистические задержки;
- проблемы с источником данных;
- аварийные/плановые остановки.

Формат:

- компактный список последних/самых значимых событий;
- время;
- актив;
- стадия;
- длительность;
- влияние;
- связь с отклонением.

Этот блок не должен быть на первом месте. Он нужен как объясняющий слой после план-факта и списка отклонений.

## Что не выносить на главную

- "Ресурсная база" как основной блок ежедневной главной.
- Полные таблицы всех KPI каждой стадии.
- Стратегические запасы/ресурсы, если пользователь решает оперативную задачу.
- Технические коэффициенты без влияния на план.
- Повторяющиеся карточки с одинаковым приоритетом.
- Необъясненные "узкие места", если нет данных о причинах.

Ресурсную базу можно вернуть как отдельный раздел или вкладку, если задача главной расширится до стратегического обзора.

## Правила статусов

Статусы:

- "В норме" - зеленый;
- "Отклонение" - желтый/amber;
- "Критично" - красный;
- "Нет данных" - серый;
- "Не классифицировано" - нейтральный.

Статус должен учитывать направление нормы:

- для добычи, переработки, производства и извлечения обычно больше лучше;
- для простоев обычно меньше лучше;
- для КИО/КТГ обычно больше лучше;
- для отклонений содержания Au нужен контекст плана и диапазона;
- для горной массы и вскрыши статус зависит от производственного плана, а не от абсолютного роста.

## Визуальное направление

Основное направление: light SaaS dashboard по приложенным референсам.

Правила:

- светлая рабочая область;
- белые или почти белые карточки;
- мягкий серый фон;
- крупные радиусы `16-20px` для карточек;
- компактные pills для дельт и статусов;
- зеленый, красный и amber использовать только семантически;
- желтый брендовый акцент использовать дозированно;
- график должен быть главным смысловым элементом, не декором;
- таблицы делать легкими, с row separators, а не тяжелой сеткой;
- не вкладывать карточки в карточки;
- не перегружать первый экран одинаковыми плитками.

Темный вариант можно держать как future operator mode, но не как основную тему для первой версии.

## Desktop layout

Первый экран:

1. Header: название, период, фильтр активов, время обновления.
2. KPI-лента в одну строку.
3. Две колонки:
   - большой `PlanFactChart`;
   - `DeviationReasonsList`.
4. Ниже:
   - `ProductionFlow`;
   - `AssetPerformanceTable`.
5. Внизу или справа на широких экранах:
   - `ProductionEventsList`.

## Mobile layout

Порядок блоков:

1. Header и период.
2. Итоговый KPI "Золото произведено".
3. KPI-лента горизонтальным scroll или 2xN.
4. Отклонения и причины.
5. План-факт график.
6. Активы.
7. Производственный поток.
8. События.

На мобильном важнее проблемы и статус, чем широкий график.

## Состояния

### Loading

Skeleton должен повторять будущую структуру: KPI-лента, график, список отклонений, таблица активов.

### Empty

Показывать, чего именно нет:

- нет производственных данных;
- нет данных по причинам;
- нет событий;
- нет данных по конкретному активу.

### Error

Ошибки должны быть частичными:

- если не загрузился список событий, не ломать весь dashboard;
- если не загрузился главный план-факт, показывать KPI и отклонения;
- если нет данных по причине, показывать "причина не классифицирована".

### No data

"Нет данных" является отдельным статусом и может попадать в `DeviationReasonsList`, если метрика критична для управления.

## Переходы

- KPI-карточка ведет в детализацию метрики.
- Точка графика ведет в детализацию периода/смены.
- Отклонение ведет в карточку причины или метрики.
- Актив ведет в страницу актива.
- Событие ведет в журнал событий или связанный производственный показатель.

## Критерии готовности дизайна

- Главная за 10-20 секунд отвечает, выполняется ли план.
- Видно не только отклонение, но и вероятную причину.
- Активы сравнимы между собой без открытия детализации.
- Главный график объясняет динамику, а не просто украшает экран.
- Визуальный стиль ближе к приложенным Muzli/Zajno референсам, а не к сайту Polyus.
- Polyus-данные используются только как доменная опора для метрик и производственной структуры.
- Нет лишнего стратегического блока "Ресурсная база" на первом экране.
- Нет абстрактного блока "Узкие места" без данных о причинах.
