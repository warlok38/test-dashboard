# Дизайн-спека: дашборд планерки по Figma и новым endpoint-ам

## Контекст

Нужно реализовать экран дашборда планерки по Figma-макету `rOZ3WvbM5SnMVfqylOZPY8`, node `492:2285`. Макет является источником правды по компоновке, размерам, текстам, цветовой роли элементов и визуальной плотности.

Предыдущая концепция executive overview заменяется на точное повторение приложенного макета: графитовый topbar и sidebar, светлая рабочая область, раскрытая плашка "Добыча", сетка месторождений, свернутые секции "Минеральные ресурсы" и "Обогащение", нижний footer.

## Цель

Пользователь открывает главную страницу и видит оперативную планерочную сводку по всем месторождениям. Из sidebar он может перейти на отдельную страницу месторождения с тем же визуальным языком и графиком план-факт.

## Основные решения

- Главная страница `/` показывает общий режим `all`.
- Страницы месторождений имеют короткие URL: `/olimpiada`, `/natalka`, `/blagodatnoe`, `/kuranah`, `/suhoy-log`.
- Endpoint `/gtk` возвращает только русские строковые названия месторождений.
- Для URL используется временный локальный словарь соответствия русского названия и slug.
- Endpoint `/summary` используется для главной и детальной страницы.
- Endpoint `/graph` используется на детальной странице месторождения.
- Фильтр БЕ удаляется из header и переносится в sidebar как раздел "Месторождения".
- Раздел "Месторождения" в sidebar раскрыт по умолчанию.

## Рассмотренные подходы

### Вариант A: оставить старую структуру `/production-stages`

Старые маршруты и текущие production-stage страницы можно было бы стилизовать под новый макет. Этот путь требует меньше новых route-ов, но плохо соответствует требованию коротких URL и новому backend-контракту.

### Вариант B: GTK как основной раздел навигации

Главная `/` показывает `/summary` без `gtk`, а каждый пункт sidebar ведет на короткий route месторождения. Внутри route slug мапится на русское название, которое передается в `/summary` и `/graph`.

Это выбранный подход.

### Вариант C: короткие URL плюс query-параметр

Например `/olimpiada?gtk=Олимпиада`. Это упрощает передачу backend-значения, но дублирует состояние в URL и делает навигацию менее чистой.

## Figma-ориентиры

Фрейм `492:2285` имеет размер `1994x1018`.

Основная структура:

- topbar: `40px`;
- sidebar: `280px`;
- main wrapper: `1714px`;
- footer: `32px`;
- sidebar начинается под topbar;
- контент имеет отступ `20px`;
- карточки и плашки имеют небольшие скругления и спокойную промышленную плотность.

Текстовые элементы из макета сохраняются:

- бренд: `ЦИФРОВОЙ ГОК`;
- приветствие: `Доброе утро, Ярослав Сергеевич`;
- sidebar: `Главная`, `Месторождения`, список GTK, `Настройки`;
- breadcrumb: `ПРОИЗВОДСТВО · 26 июня 2026`;
- секции: `Добыча`, `Месторождения`, `Минеральные ресурсы`, `Обогащение`;
- бейджи: `В норме`, `Критично`, `Отклонение`, `Виджет в работе`;
- footer: `Цифровой ЗИФ`, `Сервис LLM`, `Запросов`, `Пользователей`.

## API-контракты

### `/gtk`

Возвращает массив строк:

```ts
type GtkResponse = string[]
```

Пример:

```ts
;['Олимпиада', 'Благодатное', 'Наталка', 'Куранах', 'Сухой Лог']
```

Так как backend отдает только строки, frontend временно использует локальный словарь:

```ts
const GTK_SLUG_BY_NAME = {
  Олимпиада: 'olimpiada',
  Благодатное: 'blagodatnoe',
  Наталка: 'natalka',
  Куранах: 'kuranah',
  'Сухой Лог': 'suhoy-log'
} as const
```

Обратный словарь нужен для route-ов:

```ts
const GTK_NAME_BY_SLUG = {
  olimpiada: 'Олимпиада',
  blagodatnoe: 'Благодатное',
  natalka: 'Наталка',
  kuranah: 'Куранах',
  'suhoy-log': 'Сухой Лог'
} as const
```

Если `/gtk` вернет неизвестное название, в первой итерации оно не отображается в sidebar, чтобы не создавать нестабильные URL.

### `/summary`

Параметры:

```ts
type SummaryQuery = {
  date_from: string
  date_to: string
  gtk?: string
}
```

Для общей главной `gtk` не передается или передается пустым значением. Backend трактует это как `all`.

Примеры:

```txt
GET /summary?date_from=2026-06-26&date_to=2026-06-26
GET /summary?date_from=2026-06-26&date_to=2026-06-26&gtk=Олимпиада
```

Тип ответа:

```ts
type AlarmSummaryResponse = {
  production_date_from: string
  production_date_to: string
  shift: number
  total_critical: number
  total_warning: number
  total_incidents: number
  by_stage: Record<string, StageSummary>
}

type StageSummary = {
  display_name: string
  critical: number
  warning: number
  info: number
  cards: SummaryIndicatorCard[]
}

type SummaryIndicatorCard = {
  indicator_name: string
  plan_value: number
  fact_value: number
  deviation_pct: number
  severity: string
  measure_unit: string
  details: SummaryIndicatorDetail[]
}

type SummaryIndicatorDetail = {
  gtk_or_zif: string
  plan_value: number
  fact_value: number
  deviation_pct: number
  severity: string
}
```

### `/graph`

Параметры:

```ts
type GraphQuery = {
  indicator: string
  date?: string
  date_from?: string
  date_to?: string
  gtk: string
}
```

Ответ:

```ts
type GraphPoint = {
  date: string
  fact: number | null
  plan: number | null
}
```

`null` в `fact` или `plan` означает отсутствие точки данных. На графике это не превращается в `0`; нужно показать разрыв линии или пустое состояние для серии.

## Главная страница `/`

### Запросы

- `GET /gtk`
- `GET /summary` с `date_from` и `date_to`, без `gtk`.

### Layout

Главная повторяет Figma:

1. Topbar.
2. Sidebar.
3. Main content.
4. Footer.

Текущий `ContentHeader` не должен показывать фильтр БЕ. Фильтрация по месторождениям переносится в sidebar.

### Sidebar

Sidebar содержит:

- дату и время;
- ссылку `Главная`;
- раскрытый раздел `Месторождения`;
- пункты из `/gtk`, пропущенные через локальный mapper;
- ссылку `Настройки`.

Пункт `Главная` ведет на `/`.

Пункты месторождений ведут на короткие URL:

- `Олимпиада` -> `/olimpiada`;
- `Наталка` -> `/natalka`;
- `Благодатное` -> `/blagodatnoe`;
- `Куранах` -> `/kuranah`;
- `Сухой Лог` -> `/suhoy-log`.

### Секция "Добыча"

Источник: `summary.by_stage.mining`.

Если ключ `mining` отсутствует, можно искать стадию по `display_name === 'Добыча'`. Если стадия не найдена, показывается empty-state внутри плашки "Добыча".

Заголовок:

- текст: `Добыча`;
- бейдж вида `3/4` как в макете.

Для бейджа в первой итерации используется правило:

```ts
healthy = cards.length - count(cards where severity is critical or warning)
total = cards.length
```

Карточки KPI:

- берутся из `stage.cards`;
- название: `indicator_name`;
- единица: `measure_unit`;
- основное значение: `fact_value`;
- подпись: `deviation_pct` + `к плану`;
- цвет подписи зависит от `severity`;
- порядок сохраняется как пришел от backend.

### Блок "Месторождения"

Источник: `summary.by_stage.mining.cards[].details`.

Данные группируются по `gtk_or_zif`. Для каждого месторождения строится карточка как в макете:

- заголовок;
- статусный бейдж;
- сетка показателей;
- значение = `fact_value`;
- единица берется из родительской карточки `measure_unit`;
- подпись = `deviation_pct` + `к плану`.

Статус карточки:

- если есть `critical` -> `Критично`;
- иначе если есть `warning` -> `Отклонение`;
- иначе `В норме`.

Карточка месторождения может быть ссылкой на короткий route, если `gtk_or_zif` есть в mapper.

### Секции "Минеральные ресурсы" и "Обогащение"

В этой итерации они статичны:

- свернутый вид как в Figma;
- бейдж `Виджет в работе`;
- стрелка раскрытия визуально присутствует;
- интерактивное раскрытие можно не делать, если это не требуется для точного поведения.

### Footer

Левая часть:

- `Цифровой ЗИФ` - ссылка;
- `Сервис LLM` - ссылка.

Правая часть:

- статичный текст;
- группы `Запросов` и `Пользователей`;
- если число отсутствует в будущем контракте, показывать `-`, например `Сегодня: -`.

## Детальная страница месторождения

### URL

Route создается по короткому slug:

```txt
/:gtkSlug
```

Допустимые slug проверяются через `GTK_NAME_BY_SLUG`. Неизвестный slug ведет в `notFound()`.

### Запросы

Для `/olimpiada`:

```txt
GET /summary?date_from=2026-06-26&date_to=2026-06-26&gtk=Олимпиада
GET /graph?indicator=<first_indicator>&date_from=2026-06-26&date_to=2026-06-26&gtk=Олимпиада
```

`first_indicator` - первый `indicator_name` из `summary.by_stage.mining.cards`.

Если в `mining.cards` нет карточек, `/graph` не вызывается, а область графика показывает empty-state.

### Содержимое

Страница содержит:

- breadcrumb/заголовок под выбранное месторождение;
- плашку `Добыча`;
- плашку `Минеральные ресурсы`;
- плашку `Обогащение`;
- график план-факт.

Плашки используют тот же компонентный стиль, что и главная, но данные фильтруются backend-ом через `gtk`.

График строится по `/graph`:

- `fact` и `plan` - две серии;
- ось X - `date`;
- `null` не заменяется на `0`;
- loading/error/empty состояния обрабатываются отдельно.

## Архитектура

Сохраняется soft Feature-Sliced Design:

- `src/app` - тонкие маршруты `/` и `/:gtkSlug`;
- `src/widgets` - layout, sidebar, home dashboard, gtk detail dashboard;
- `src/entities` - типы, api и мапперы доменных данных;
- `src/features` - интерактивные переключатели графика, если они появятся;
- `src/shared` - UI primitives, форматтеры, constants.

Рекомендуется добавить новую entity:

```txt
src/entities/production-summary
```

В нее входят:

- RTK Query endpoints `/gtk`, `/summary`, `/graph`;
- типы backend-контрактов;
- helpers для статусов, форматирования отклонений и группировки details по GTK;
- временный mapper `gtkName <-> slug`.

Так новые backend-контракты не смешиваются со старой entity `production-stage`.

UI лучше разделить так:

```txt
src/widgets/planning-dashboard
src/widgets/gtk-detail-dashboard
```

Или одним модулем:

```txt
src/widgets/production-summary-dashboard
```

Рекомендуемый вариант - один общий модуль `production-summary-dashboard`, потому что главная и детальная страницы используют одни и те же карточки, плашки и форматтеры.

## Состояния

### Loading

Показывать skeleton по форме макета: sidebar может загрузить GTK отдельно, content - summary отдельно, graph - отдельно.

### Error

Если `/summary` не загрузился, основная область показывает ошибку внутри content, но layout остается.

Если `/gtk` не загрузился, sidebar показывает `Месторождения` без пунктов или compact error-state.

Если `/graph` не загрузился, детальная страница сохраняет summary и показывает ошибку только в области графика.

### Empty

Если `by_stage.mining` отсутствует или `cards` пустой:

- плашка `Добыча` остается на месте;
- внутри показывается понятное empty-state;
- блок `Месторождения` не строится из пустых details.

## Responsive

Desktop должен точно соответствовать Figma-макету при ширине около `1994px`.

Для меньших экранов:

- sidebar остается drawer/mobile-nav по текущей проектной логике;
- KPI карточки переносятся в 2 или 1 колонку;
- карточки месторождений переходят из 3 колонок в 2 и затем в 1;
- footer может переносить правую группу под левую;
- текст не должен выходить за карточки и бейджи.

## Не входит в итерацию

- полноценные данные для "Минеральные ресурсы" и "Обогащение";
- backend-driven slug;
- авторизация и реальные данные пользователя topbar;
- редактирование footer-счетчиков;
- сложный выбор indicator на графике, кроме дефолта по первому `indicator_name`;
- изменение старых детальных страниц `/production-stages`.

## Критерии готовности

- Создана отдельная ветка задачи.
- Главная визуально повторяет Figma node `492:2285`.
- `/gtk` наполняет sidebar.
- `/summary` без `gtk` наполняет главную в режиме all.
- `/summary` с `gtk` наполняет страницу месторождения.
- `/graph` строит план-факт график на странице месторождения.
- Короткие URL месторождений работают.
- Фильтр БЕ отсутствует в header и представлен в sidebar как "Месторождения".
- Footer соответствует макету и показывает `-` для отсутствующих чисел.
- `npm.cmd run lint` и `npm.cmd run build` проходят после реализации.
